    Evals are how you find out your agent is broken
    before your users do.

    Why evals matter

    Three reasons standard testing breaks for AI.

    01

    LLMs are non-deterministic.

    Same input → different output. Your assertEqual is dead.

    02

    Failure is high-dimensional.

    "Wrong" can mean hallucinated, off-tone, leaks PII, ignores tools, loops forever, or just subtly worse than yesterday.

    03

    Vibes don't scale.

    You can eyeball 20 traces. You cannot eyeball 20,000. If you can't eyeball them, you don't know what your system does.

    What an eval actually is

    An eval is three things.

    1. A dataset   — inputs that look like real user traffic
2. A grader    — a function: (input, output) → score
3. A runner    — runs the system on the dataset, applies the grader, reports

    Everything else is variations on these three.

    "Write evals for errors you discover,
    not errors you imagine."

    — Hamel Husain

    What instrumentation looks like.

    from opentelemetry import trace
tracer = trace.get_tracer("job-search-agent")

@tracer.start_as_current_span("agent.run")
def run_agent(user_id: str) -> AgentResult:
    span = trace.get_current_span()
    span.set_attribute("agent.user_id", user_id)

    posts = fetch_linkedin_feed(user_id)
    with tracer.start_as_current_span("llm.extract") as s:
        result = extract_job_info(post)
        s.set_attribute("llm.is_matched", result["isMatched"])
        s.set_attribute("llm.company",    result["company"])
        s.set_attribute("llm.role",       result["role"])

    If your agent doesn't have spans like this today, that's the first thing you do Monday.

    The five pillars

    What you actually grade.

    01
    Task success
    Did the user get what they asked for?
    02
    Trajectory
    Sensible path, or a wandering mess?
    03
    Tool use
    Right tool, right args, right order?
    04
    Safety
    No PII leaks, destructive cmds, jailbreaks?
    05
    Cost & latency
    Usable in production at scale?

    Pick the right grader.

    GraderWhen to useCostReliability
    Code / assertionStructured outputs, tool args, JSON schemasFreeHighest
    LLM-as-judgeSubjective quality, tone, helpfulness$$Medium · needs calibration
    Human reviewAnything novel, ground truth, calibration$$$Highest · doesn't scale

    Use code when you can, models when you must, humans to calibrate.

    Code

    Every pillar fits in ~3 lines of code.

    # 1. Task success — all three fields correct
def grade_extraction(case, result) -> bool:
    return result["isMatched"] == case.expected and result["company"] == case.company and result["role"] == case.role

# 2. Trajectory — pre-filter ran before LLM calls
def grade_trajectory(trace) -> bool:
    return trace.llm_calls <= 25  # filter should cut ~120 posts to <25

# 3. Tool use — no notification sent when isMatched is false
def grade_no_spam(trace) -> bool:
    return all(c.args["isMatched"] for c in trace.tool_calls if c.tool == "send_telegram")

# 4. Safety — notification goes to the right user
def grade_user_isolation(trace) -> bool:
    return all(c.args["user_id"] == trace.user_id for c in trace.tool_calls if c.tool == "send_telegram")

# 5. Cost — shouldn't cost more than $0.02 per user run
def grade_cost(trace) -> bool:
    return trace.total_cost_usd < 0.02

    Every grader returns a boolean. Binary scoring is non-negotiable.

    Build the eval suite

    The four-step playbook.

    1. Build the dataset from your traces (real failures, real posts)
2. Pick the right grader for each criterion (code > LLM > human)
3. Wire into CI — run on every LLM or prompt change
4. Close the loop — feed prod failures back into the dataset

    Pitfalls

    Four things I wish I'd known.

    01
    Eval-driven development is mostly wrong. Look at data first; write evals for failures you've actually seen.
    02
    Don't outsource your annotations. Domain expertise is the bottleneck.
    03
    Likert scales lie. A 1–5 score from a judge is two bits of signal cosplaying as five. Use binary + a reason.
    04
    More metrics ≠ better evals. 5 well-chosen metrics beat 30 noisy ones.

    The checklist

    Monday morning. Photograph this slide.

    Add OTel-style tracing if you don't have it.
    Pull 50 real traces from your agent. Read all of them.
    Group failures into 3–5 buckets. That's your eval taxonomy.
    Build a 100-example dataset weighted by those buckets.
    Pick ONE grader per bucket (code > LLM-judge > human).
    Run each task k=5 times. Report pass^5, not pass@1.
    Wire the suite into CI. Block merges on regression.
    Calibrate any LLM-judge against 50 human-labeled examples.
    Sample 1% of prod, run the same judges, watch for drift.
    Lock in a versioned baseline so you can A/B the next model upgrade.
